# 协作模式详细说明

[总体流程]

```
用户指令
    ↓
解析指令类型 + 集数
    ↓
读取 .agent-state.json
    ↓
组织信息（projectConfig + userPreferences + contextFiles）
    ↓
路由到子 Agent（storyboard-artist/director/animator）
    ↓
子 Agent 调用对应 SKILL
    ↓
SKILL 读取 common/ 模块，执行具体逻辑
    ↓
写入输出文件
    ↓
更新 .agent-state.json
    ↓
返回制片人
    ↓
【关键步骤】制片人执行用户确认（CP1-CP5）
    ├─ 通过 → 创建对应类型快照 → 进入下一阶段
    ├─ 修改 → 根据反馈修改 → 重新审核
    ├─ 重生成 → 重新执行生成 → 重新审核
    └─ 回滚 → 恢复到历史快照 → 继续/暂停
    ↓
阶段完成

[角色职责]

| 角色 | 职责 | 不需要做 |
|------|------|----------|
| **制片人** | 协调、路由、组织信息、**执行用户确认** | 执行具体业务逻辑 |
| **storyboard-artist** | 调用 film-storyboard-skill 生成节拍/九宫格/四宫格 | 记忆详细步骤 |
| **director** | 调用 storyboard-review-skill 执行审核 | 重新实现审核逻辑 |
| **animator** | 调用 animator-skill 生成动态提示词 | 记忆动态生成逻辑 |

[子 Agent 详细配置]

子 Agent 只做轻量路由，实际工作由 SKILL 完成：

| 子 Agent | 配置文件 | 调用 SKILL | 说明 |
|----------|----------|------------|------|
| storyboard-artist | `agents/storyboard-artist.md` | film-storyboard-skill | 节拍拆解/九宫格/四宫格 |
| director | `agents/director.md` | storyboard-review-skill | 审核流程 |
| animator | `agents/animator.md` | animator-skill | 动态提示词生成 |
| router | `agents/router.js` | - | 指令路由 |

[阶段流转]

```
1. 用户输入剧本 → /breakdown → storyboard-artist → film-storyboard-skill
   → outputs/beat-breakdown-{集数}.md

2. 节拍拆解表 → /beatboard → storyboard-artist → film-storyboard-skill
   → outputs/beat-board-prompt-{集数}-board*.md

3. 九宫格提示词 → /sequence → storyboard-artist → film-storyboard-skill
   → outputs/sequence-board-prompt-{集数}.md

4. 四宫格提示词 → /motion → animator → animator-skill
   → outputs/motion-prompt-{集数}.md

5. 各阶段产出 → /review → director → storyboard-review-skill
   → 审核报告
```

[异常处理]

1. 缺少前置产物时，提示用户先完成上一阶段
2. 文件不存在时，返回明确错误信息
3. 审核不通过时，返回修改建议

[项目状态检测]

收到 `/status` 或初始化时自动执行：

1. 扫描 `script/` 识别剧本文件
2. 扫描 `outputs/` 识别已完成的产物
3. 读取 `.agent-state.json` 获取完整状态
4. 根据文件存在性判断进度

[进度判断规则]

```
无 beat-breakdown → [节拍拆解阶段]
有 beat-breakdown，无 beat-board → [九宫格阶段]
有 beat-board，无 sequence → [四宫格阶段]
有 sequence，无 motion → [动态提示词阶段]
全部都有 → [完成]
```

[制片人用户确认流程]

制片人在每个阶段完成后必须执行以下用户确认流程：

```
返回制片人
    ↓
确定当前确认点（CP1-CP5）
    ↓
展示标准确认UI（按 user-confirmation-protocol.md 规范）
    ↓
等待用户选择：
├─ [通过] ✅ → 创建 → 进入下一阶段
├─ [快照修改] ✏️ → 收集修改意见 → 标记状态为"待修改" → 等待修改后重审
├─ [重生成] 🔄 → 创建快照 → 重新调用子Agent生成
└─ [回滚] ⏪ → 展示快照列表 → 用户选择 → 回滚 → 重新审核
    ↓
确认完成 → 更新 .agent-state.json
```

**确认点与产出对应关系：**

| 确认点 | 产出物 | 快照类型 |
|--------|--------|----------|
| CP1 | 节拍拆解表 | 完整快照 |
| CP2 | 九宫格提示词 | 分板快照 |
| CP3 | 四宫格提示词 | 分组快照 |
| CP4 | 动态提示词 | 参数快照 |
| CP5 | 所有产出 | 完整快照 |

**详细确认UI模板：**

```
[阶段名称] 确认 (CP[编号])
──────────────────────────────
[██████████] 100%

关键产出概览
├─ 产出数量：[N]个
├─ 质量评分：[评分]/5
├─ 审核状态：[通过/需修订/不通过]
└─ 主要特点：[描述]

操作选项
├─ [通过] ✅
├─ [修改] ✏️
├─ [重生成] 🔄
└─ [回滚] ⏪
```

**执行时机：**
- `/breakdown` 完成后 → CP1
- `/beatboard` 完成后 → CP2
- `/sequence` 完成后 → CP3
- `/motion` 完成后 → CP4
- 所有阶段完成后 → CP5

[快照管理]

详见 `common/snapshot-management.md`

每个确认点自动创建对应类型的快照：
- CP1: 完整快照
- CP2: 分板快照
- CP3: 分组快照
- CP4: 参数快照
- CP5: 完整快照

**快照操作：**
- `/snapshot list` - 列出快照
- `/snapshot preview <id>` - 预览快照
- `/snapshot rollback <id>` - 回滚到快照
- `/snapshot create "备注"` - 创建手动快照
