# ANINEO V2.0 模块 2 重构总结

> **重构日期**: 2025-01-22
> **重构目标**: 取消 Beatboard 固定 4 格限制，改为基于关键帧权重（🔴/🟠/🟡）的动态展示逻辑
> **核心原则**: 确保"高光"全显，"支柱"择优，"水下"全量。展示规模应忠实反映剧本的节奏起伏

---

## ✅ 已完成的任务

### 1. 修改 film-storyboard-skill/SKILL.md - 动态选择算法升级 ✅
**内容**: 步骤3（执行关键帧选择）扩展

**新增内容**:
- **动态展示策略（Dynamic Anchor Strategy）**:
  - 强制展示（🔴 特级 / 🟠 高级）：所有红色或橙色关键帧必须作为视觉锚点展示
  - 弹性展示（🟡 中级）：根据节拍节奏筛选，关键动机转折则选入展示名单
  - 剔除展示（🔵 普通 / ⚪ 次要）：原则上不出现在 Beatboard 展示层，仅作为"水下全量数据"存在于 scene-breakdown.json 中
- **展示规模控制**:
  - 展示数量不再固定为 4，而是根据该场戏的实际权重分布动态生成（可能是 3 个，也可能是 6 个）
  - 展示规模在 3-9 格之间，确保视觉信息的"信噪比"最高
- **确保原则**:
  - "高光"全显：所有 🔴 特级和 🟠 高级关键帧必须展示
  - "支柱"择优：🟡 中级关键帧根据节拍节奏筛选
  - "水下"全量：🔵 普通和 ⚪ 次要关键帧仅存在于 scene-breakdown.json 中

### 3. 同步修改 AGENTS.md - 审核逻辑与职责 ✅
**内容**: Director 审核准则扩展

**新增审核项**:

#### 审核项：关键帧覆盖度【V2.0 模块 2 新增】

**审核问题 1**：所有的 🔴 和 🟠 关键帧是否都已在 Beatboard 中展示？
- **检查方法**：对比 scene-breakdown-{集数}.json 中的关键帧等级和 Beatboard 展示的格子
- **FAIL 条件**：任何 🔴 或 🟠 关键帧未在展示层
- **PASS 条件**：所有 🔴 和 🟠 关键帧都在展示层

**审核问题 2**：展示出的 🟡 关键帧是否真的起到了支撑叙事节奏的作用？
- **检查方法**：验证每个展示的 🟡 关键帧的"选入原因"字段
- **FAIL 条件**：🟡 关键帧无明确选入原因，或选入原因不合理
- **PASS 条件**：所有 🟡 关键帧都有合理的选入原因（节奏支撑/动机转折/空间连贯/轴线跨越）

#### 审核项：冰山完整性【V2.0 模块 2 新增】

**审核问题**：虽然展示数量有变，但 scene-breakdown.json 中是否依然完整包含了所有 NDI 拆解出的原子镜头？
- **检查方法**：统计 scene-breakdown-{集数}.json 中的 shots 数量，对比节拍拆解表的节拍数量
- **FAIL 条件**：shots 数量与 NDI 拆解出的原子镜头数量不一致
- **PASS 条件**：全量层完整包含所有 NDI 拆解出的原子镜头

#### 审核项：展示规模合理性【V2.0 模块 2 新增】

**审核问题**：展示数量是否忠实反映了剧本的节奏起伏？
- **检查方法**：分析展示数量（3-9格）与剧本节奏的匹配度
- **FAIL 条件**：展示数量与剧本节奏明显不符（如：高密度动作场景仅展示 3 格）
- **PASS 条件**：展示数量与剧本节奏成正比

### 4. 修改 layout-calculator.md - 支持动态布局 ✅
**内容**: 更新为支持基于关键帧权重的动态布局

**新增内容**:
- **三级展示规则**:
  - 🔴 特级（≥3.0）：强制展示，高密度动作场景 7-9 格
  - 🟠 高级（2.0-2.9）：强制展示，中密度冲突场景 5-7 格
  - 🟡 中级（1.0-1.9）：弹性展示，低密度氛围场景 3-5 格
  - 🔵 普通（0.5-0.9）：剔除展示
  - ⚪ 次要（<0.5）：剔除展示

- **🟡 中级关键帧筛选规则**:
  - 选入条件：节奏支撑（跨度过大）、动机转折、空间连贯
  - 剔除条件：高级关键帧间距合理、无明显动机转折、不是轴线跨越

- **动态布局规范**:
  | 展示数量 | 推荐布局 | 说明 | Markdown 排版 |
  |----------|----------|------|-------------|
  | 3 格 | 1×3 单列布局 | 线性叙事，节奏缓慢 | 单列，每格占一行 |
  | 4 格 | 2×2 网格布局 | 平衡叙事，节奏适中 | 2 行 2 列网格 |
  | 5 格 | 2×3 网格布局（最后一行 1 格居中）| 轻微节奏起伏 | 2 行 3 列，第 2 行居中 |
  | 6 格 | 2×3 网格布局 | 标准叙事节奏 | 2 行 3 列网格 |
  | 7 格 | 2×4 网格布局（最后一行 3 格居中）| 中高节奏密度 | 2 行 4 列，第 2 行居中 |
  | 8 格 | 2×4 网格布局 | 高节奏密度 | 2 行 4 列网格 |
  | 9 格 | 3×3 网格布局 | 最高节奏密度 | 3 行 3 列网格 |

- **布局选择建议算法**:
  ```python
  def suggest_layout_v2(shots_data):
      # 统计关键帧等级分布
      level_distribution = {
          "🔴": count_by_level(shots_data, "🔴"),
          "🟠": count_by_level(shots_data, "🟠"),
          "🟡": count_by_level(shots_data, "🟡")
      }
      
      # 强制展示所有 🔴 和 🟠
      mandatory_count = level_distribution["🔴"] + level_distribution["🟠"]
      
      # 弹性展示 🟡（根据节拍节奏筛选）
      optional_count = select_optional_yellow(shots_data, level_distribution["🟡"])
      
      # 计算展示数量
      display_count = mandatory_count + optional_count
      
      # 限制在 3-9 范围内
      display_count = max(3, min(9, display_count))
      
      # 建议布局
      if display_count <= 3:
          layout_type = "1×3 单列布局"
      elif display_count == 4:
          layout_type = "2×2 网格布局"
      elif 5 <= display_count <= 7:
          layout_type = "2×3 网格布局"
      else:
          layout_type = "2×4 或 3×3 网格布局"
      
      return {
          "display_count": display_count,
          "layout_type": layout_type,
          "layout_description": 布局描述
      }
  ```

### 5. 在 data-validator.md 中新增 Beatboard 动态布局验证 ✅
**内容**: 新增 BeatboardDynamicLayoutValidator 类

**新增验证方法**:

#### validate_keyframe_coverage() - 关键帧覆盖度验证【V2.0 模块 2 核心审核项】

- **检查**: 所有的 🔴 和 🟠 关键帧是否都已在 Beatboard 中展示？
- **方法**: 对比 scene-breakdown-{集数}.json 中的关键帧等级和 Beatboard 展示的格子
- **FAIL 条件**: 任何 🔴 或 🟠 关键帧未在展示层
- **PASS 条件**: 所有 🔴 和 🟠 关键帧都在展示层

#### validate_yellow_frames_rationale() - 🟡 中级关键帧合理性验证

- **检查**: 展示出的 🟡 关键帧是否真的起到了支撑叙事节奏的作用？
- **方法**: 验证每个展示的 🟡 关键帧的"选入原因"字段
- **FAIL 条件**: 🟡 关键帧无明确选入原因，或选入原因不合理
- **PASS 条件**: 所有 🟡 关键帧都有合理的选入原因（节奏支撑/动机转折/空间连贯/轴线跨越）

#### validate_display_size_reasonableness() - 展示规模合理性验证

- **检查**: 展示数量是否忠实反映了剧本的节奏起伏？
- **方法**: 分析展示数量（3-9格）与剧本节奏的匹配度
- **场景 1**: 高密度动作场景（高 NDI 分数 ≥3），展示数量应较多（7-9 格）
- **场景 2**: 低密度氛围场景（低 NDI 分数 <1.5），展示数量应较少（3-5 格）
- **FAIL 条件**: 展示数量与剧本节奏明显不符
- **PASS 条件**: 展示数量与剧本节奏成正比

#### validate_iceberg_completeness() - 冰山完整性验证

- **检查**: 虽然展示数量有变，但 scene-breakdown.json 中是否依然完整包含了所有 NDI 拆解出的原子镜头？
- **方法**: 统计 scene-breakdown-{集数}.json 中的 shots 数量，对比节拍拆解表的节拍数量
- **FAIL 条件**: shots 数量与 NDI 拆解出的原子镜头数量不一致
- **PASS 条件**: 全量层完整包含所有 NDI 拆解出的原子镜头

#### validate_anchor_metadata_completeness() - 锚点元数据完整性验证

- **检查**: 每个展示的锚点是否包含完整的元数据？
- **方法**: 检查是否有"关键帧等级"、"选入原因"、"镜头动机"、"NDI 分数"
- **FAIL 条件**: 缺少任何必填字段
- **PASS 条件**: 所有锚点元数据完整
